# 洛谷自动评测浏览器扩展 —— 需求规格说明

> 版本：0.1.0 | 状态：待验收 | 创建日期：2026-06-18

---

## 1. 立项背景

当前仓库中 OJ 题解的测试流程为：终端编写代码 → `g++` 编译 → 手动运行并肉眼比对输出。当需要将代码提交到洛谷进行在线评测（AC 验证）时，必须手动打开浏览器、定位题目、粘贴代码、点击提交、等待结果——这一流程在批量处理多道题目时效率极低。

本扩展的目标是将「终端→浏览器→评测→结果回传」的链路自动化，使用者只需在终端中执行一条命令即可完成大批量代码的在线评测。

---

## 2. 系统概述

本系统由三个部分构成：

| 组件           | 运行位置                 | 职责                                                    |
| -------------- | ------------------------ | ------------------------------------------------------- |
| **CLI 工具**   | 用户终端（WSL / Linux）  | 扫描本地代码文件、发送至本地服务、展示评测结果          |
| **本地服务**   | 用户本机（全局守护进程） | 接收 CLI 请求、桥接浏览器扩展、调度任务队列、缓存结果   |
| **浏览器扩展** | Chromium 内核浏览器      | 建立 WebSocket 服务端，接收任务、模拟用户操作、回传结果 |

通信链路：`CLI --HTTP--> 本地服务 --WebSocket (client)--> 浏览器扩展 (server)`

## 3. 功能需求

### 3.1 CLI 工具 (`luogu-cli`)

#### 3.1.1 代码提交

- **F-001 单文件提交**：用户指定一个 `.cpp` 文件路径，将其提交至对应洛谷题目进行评测。
- **F-002 批量提交**：用户指定一个目录，自动扫描其中所有 `.cpp` 文件并批量提交。
  - 自动从文件名解析题目 ID（如 `P1001.cpp` → 题目 P1001）。
  - 支持 `--include` / `--exclude` 过滤文件。
- **F-003 提交队列**：同时提交多个文件时，按序排队，可配置最大并发数（默认 1，避免洛谷限流）。
- **F-004 重试机制**：评测失败（非 WA 的业务错误，如网络超时）可自动重试，可配置重试次数（默认 3 次）。

#### 3.1.2 结果展示

- **F-005 实时进度**：显示当前评测队列的进度条（如 `[3/12] P1001 评测中...`）。
- **F-006 评测结果摘要**：每道题目评测完成后显示：
  - 题目编号
  - 评测状态（AC / WA / TLE / RE / CE / ... 与洛谷一致）
  - 各测试点得分
  - 耗时与内存
- **F-007 批量结果导出**：支持将最终结果导出为 JSON / Markdown 表格。

#### 3.1.3 配置

- **F-008 服务地址配置**：可配置本地服务的地址与端口（默认 `127.0.0.1:9527`）。
- **F-009 登录依赖**：扩展不接管洛谷登录流程。用户需事先在浏览器中手动登录洛谷并保持 Cookie 有效，扩展复用浏览器中的已登录会话执行提交操作。

### 3.2 本地服务 (`luogu-daemon`)

#### 3.2.1 核心功能

- **F-010 CLI 接口**：提供 HTTP 接口供 CLI 工具提交任务：
  - `POST /api/tasks` — 创建评测任务
  - `GET /api/tasks/:id` — 查询任务状态与结果
  - `GET /api/tasks` — 批量查询任务
  - `DELETE /api/tasks/:id` — 取消任务
- **F-011 任务调度**：管理任务队列，按 FIFO 顺序将任务派发给浏览器扩展。
- **F-012 结果缓存**：评测结果持久化到本地 SQLite，支持历史查询。

#### 3.2.2 与浏览器扩展通信

- **F-013 通信方式**：WebSocket — 浏览器扩展作为 WS **服务端**监听本地回环地址（默认 `ws://127.0.0.1:9528`），本地服务作为 WS **客户端**连接扩展。该设计使得扩展生命周期独立于本地服务，扩展随浏览器启动即可接收任务。
- **F-014 心跳保活**：本地服务定期发送心跳（每 10s），检测浏览器扩展是否在线。扩展离线时，服务端暂停派发任务并在 CLI 给出提示。扩展重新上线后，恢复任务派发。

#### 3.2.3 状态管理

- **F-015 任务状态机**：
  ```
  pending → queued → running → completed
                            ↘ failed (可重试)
                            ↘ cancelled
  ```

### 3.3 浏览器扩展 (`luogu-extension`)

#### 3.3.1 页面自动化

- **F-016 页面导航**：根据任务中的题目 ID，自动导航至 `https://www.luogu.com.cn/problem/<pid>`。
- **F-017 代码填入**：在洛谷代码编辑器中填入提交的代码。需适配：
  - 洛谷 Monaco Editor（通过 `monaco.editor.setValue()` 或剪贴板 API 粘贴）
  - 洛谷旧版编辑器（通过 `CodeMirror.setValue()` 或 textarea 注入）
- **F-018 语言选择**：自动选择「C++17 (GCC 12)」语言选项（可通过配置修改）。
- **F-019 提交操作**：模拟点击「提交评测」按钮，并等待页面反馈。
- **F-020 结果采集**：监听提交后的评测结果更新，采集：
  - 整体状态（AC / WA / TLE / RE / MLE / CE / OLE / UKE）
  - 各子任务（subtask）/ 测试点得分
  - 最大耗时 / 最大内存

#### 3.3.2 状态指示

- **F-021 扩展图标角标**：扩展工具栏图标显示当前队列中待处理任务数。
- **F-022 页面提示**：在洛谷页面注入浮动提示条，显示当前操作状态（如「正在提交 P1001...」「P1001 评测完成: AC」）。

#### 3.3.3 安全

- **F-023 页面范围限制**：仅允许在 `www.luogu.com.cn` 域下执行自动化操作。
- **F-024 用户确认**：首次运行时，弹出确认页告知用户本扩展会自动操作页面，获得明确同意。

## 4. 非功能需求

### 4.1 性能

- **NF-001 并发限制**：默认 1 个任务并发（洛谷对提交频率有限制），可由用户调整。
- **NF-002 延迟**：本地服务到扩展的消息延迟应 < 500ms（局域网 / 本机）。

### 4.2 兼容性

- **NF-003 浏览器**：支持 Chromium 内核浏览器（Chrome >= 100, Edge >= 100, Arc, Brave 等）。
- **NF-004 系统**：CLI 与本地服务运行于 Linux / WSL，提供 x86_64 和 aarch64 版本。

### 4.3 可用性

- **NF-005 错误恢复**：浏览器标签页意外关闭后，服务端应感知并将任务重置为 `pending`。
- **NF-006 离线缓存**：当浏览器扩展未连接时，CLI 提交的任务进入队列，待扩展上线后继续执行。

### 4.4 安全

- **NF-007 本地通信**：CLI ↔ 本地服务 ↔ 浏览器扩展 的通信仅限于本地回环地址（127.0.0.1），不对外暴露端口。

## 5. 用户故事

### US-1 单题快速评测

> **作为** OIer  
> **我希望** 在终端中输入 `luogu-cli submit P1001.cpp` 后能自动完成洛谷在线评测  
> **以便** 我不用离开终端就验证代码是否能 AC。

**验收标准**：

1. 命令执行后，5 秒内本地服务确认任务已创建。
2. 浏览器自动打开 `P1001` 题目页并填入代码。
3. 评测结果在 15 秒内回传至终端（取决于洛谷评测队列）。
4. 终端显示完整的评测状态和各点得分。

### US-2 批量回归验证

> **作为** OIer  
> **我希望** 一次提交目录下所有 `.cpp` 文件进行评测  
> **以便** 重构代码后能批量验证历史题目是否仍然能 AC。

**验收标准**：

1. 支持 `luogu-cli batch source/website/www.luogu.com.cn/` 形式。
2. 自动跳过无对应洛谷题目的文件并给出警告。
3. 评测队列有序执行，终端显示整体进度。
4. 最终输出 Markdown 表格形式的汇总结果。

### US-3 首次使用引导

> **作为** 新用户  
> **我希望** 安装过程有清晰的步骤指引  
> **以便** 即使不了解系统架构也能正常使用。

**验收标准**：

1. CLI 包含 `install` 子命令：引导安装本地服务（守护进程）、加载浏览器扩展。
2. 扩展加载后，引导页展示扩展监听的 WebSocket 地址（默认 `ws://127.0.0.1:9528`），供本地服务连接使用。
3. 本地服务连接成功后，扩展图标变色表明已就绪，终端同步输出连接确认提示。

## 6. 接口协议（草案）

### 6.1 CLI → 本地服务

```
POST /api/tasks
Content-Type: application/json

{
  "pid": "P1001",
  "code": "<base64 encoded source code>",
  "language": "cpp17",
  "mode": "submit"       // "submit" | "test"
}
```

```
GET /api/tasks
Response:
{
  "tasks": [
    {
      "id": "uuid",
      "pid": "P1001",
      "status": "completed",
      "result": {
        "verdict": "AC",
        "score": 100,
        "time_ms": 45,
        "memory_kb": 1024,
        "subtasks": [...]
      },
      "created_at": "ISO8601",
      "completed_at": "ISO8601"
    }
  ]
}
```

### 6.2 本地服务 ↔ 浏览器扩展（WebSocket）

浏览器扩展在 `ws://127.0.0.1:9528` 上建立 WebSocket 服务端，本地服务作为客户端连接。

```json
// 本地服务 → 扩展: 派发任务
{
  "type": "task",
  "id": "uuid",
  "pid": "P1001",
  "code": "<base64>",
  "language": "cpp17"
}

// 扩展 → 本地服务: 状态更新
{
  "type": "progress",
  "id": "uuid",
  "step": "navigating"   // "navigating" | "filling_code" | "submitting" | "waiting_result" | "done"
}

// 扩展 → 本地服务: 评测结果
{
  "type": "result",
  "id": "uuid",
  "verdict": "AC",
  "score": 100,
  "time_ms": 45,
  "memory_kb": 1024,
  "subtasks": [
    { "index": 1, "verdict": "AC", "score": 50, "time_ms": 20, "memory_kb": 512 },
    { "index": 2, "verdict": "AC", "score": 50, "time_ms": 25, "memory_kb": 512 }
  ]
}

// 本地服务 → 扩展: 心跳
{
  "type": "ping"
}
// 扩展 → 本地服务: 心跳响应
{
  "type": "pong"
}
```

## 7. 技术选型（已决议）

| 组件       | 选用技术                                  | 决议理由                                                                           |
| ---------- | ----------------------------------------- | ---------------------------------------------------------------------------------- |
| CLI 工具   | TypeScript + Node.js                      | 与现有技术栈统一，利用 `run.sh` 的 CLI 经验                                        |
| 本地服务   | TypeScript + Node.js                      | 与 CLI、扩展端共享类型定义，降低上下文切换成本                                     |
| 浏览器扩展 | TypeScript + Chrome Extension Manifest V3 | MV3 Service Worker 可建立 WebSocket 服务端；ScriptCat 用户脚本不适合作长驻后台服务 |
| 通信协议   | WebSocket（扩展服务端 + 本地服务客户端）  | 双向实时通信，扩展生命周期独立不受客户端断连影响                                   |
| 数据持久化 | SQLite（本地服务端）                      | 轻量嵌入式，无需额外数据库服务                                                     |

## 9. 架构决策记录（ADR）

以下为已决议的架构决策，全文已按此决议更新。

| #     | 决策项   | 决议                                                    | 理由                                                                      |
| ----- | -------- | ------------------------------------------------------- | ------------------------------------------------------------------------- |
| ADR-1 | 通信方案 | WebSocket，**浏览器扩展作为服务端**，本地服务作为客户端 | 扩展生命周期独立于终端；扩展随浏览器启动即可就绪；本地服务可按需连接/断开 |
| ADR-2 | 本地服务 | 全局守护进程                                            | 避免多终端重复启动导致端口冲突；状态集中管理更可靠                        |
| ADR-3 | 洛谷登录 | 扩展不接管登录，复用浏览器已有会话                      | 降低复杂度，避免 Cookie 泄漏风险，用户手动登录即可                        |
| ADR-4 | CLI 语言 | TypeScript + Node.js                                    | 与现有技术栈一致，降低学习成本                                            |
| ADR-5 | 无头模式 | 不支持                                                  | 保持方案简洁，专注浏览器内自动化场景                                      |
